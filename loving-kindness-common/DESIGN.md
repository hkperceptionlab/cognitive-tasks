# 자애·감사 명상 (Loving-Kindness & Gratitude) — 설계·작업로그

실천(practice) 계열 **3번째** 과제(1번째=호흡세기, 2번째=바디스캔). 청소년·성인 2앱이
`loving-kindness-common/loving-kindness.js` 를 공유(`startLovingKindness({id, accent, scale, tone})`).
UI 표시명은 한국어 "자애·감사 명상"(폴더/코드 영문명은 `loving-kindness`).

## 앞 두 과제와 같은 기준 (재확인)
runTask 안 씀(엔진 시행-채점 구조 안 맞음), `core/i18n.js` 만 재사용·자체 셸, 판정 어휘 금지, '결과'가
아니라 '완료 확인', **localStorage 아무것도 안 씀**(언어선택 `cog:lang` 제외 — 스트릭·비교·기록 없음),
계열색 practice 재사용(성인 `#4A6B4D` 6.0:1 / 청소년 `#56805A` 4.6:1, 각 앱 셸이 `--lk-accent` 주입).
**엔진 코어는 이번에도 한 줄도 수정 안 함**(practice 색은 호흡세기 때 이미 등록됨).

## ★ 왜 자애명상과 감사를 하나로 합쳤나 (A안 확정)
자애명상의 5단계 확장(나→가까운 사람→모르는 사람→어려운 사람) 뒤에 감사 단계가 자연스럽게 이어지는
서사가, 꽃이 피는 시각 은유와 맞물릴 때만 의미가 산다고 판단. 분리(B안)는 이 서사를 끊어 기각.
→ 마지막 감사 단계는 **새 요소를 추가하지 않고** 이미 다 핀 꽃의 중심부가 밝아지는 것으로 표현해
"감사는 자애의 연장선"이라는 의미를 시각적으로 전달.

## ★ 바디스캔과 같은 "자기보고 없음 → 완전 자동 진행" 패턴
대상(이름)도, 감사 내용도 **입력받지 않는다**. 전부 "마음속으로 떠올려보세요" 텍스트+TTS 안내뿐.
근거: (1) 텍스트 입력이 어려운 사용자 배려, (2) 개인정보(이름 등) 입력 부담 제거, (3) 서버 없는 정적
사이트라 "저장 안 해도 프라이버시는 원래 보장" — 입력 UI 실익이 크지 않음. 그래서 세션 중 사용자 조작은
**음소거 토글 + (4단계 한정) 건너뛰기 버튼**뿐이고, 완료 화면도 점수·자기보고 지표 없이 총 시간 +
거쳐온 단계 수(5, 고정)만 사실로 표시.

## ★ 시각 예외 — 연꽃 (앞의 두 예외와 또 다른 **세 번째 근거**, 문서화 필수)
호흡세기 싱잉볼은 "장식이지만 **호흡 표시 기능**"이 근거였고, 바디스캔 실루엣은 "몸의 어느 부위인지
보여주는 것이 **안내 자체의 기능**"이 근거였다. 연꽃은 **둘 다 아니다**:

> 연꽃 = 자애의 단계 확장을 "펴지는 것"이라는 신체적/시간적 은유로 보여주는 **진행 상태의 시각적 서사**.
> 순수 장식(싱잉볼)도, 안내의 본질적 기능(실루엣)도 아닌 제3의 근거 — 단계가 넓어지는 것을 꽃이 열리는
> 것으로 은유해, 지금 어디까지 왔는지를 정서적 서사로 전달한다.

지각 계열의 '자극 무채색' 원칙은 **판단 대상 자극**에 적용되던 것이고, 이 과제엔 판단 대상 자극이 없어
(체험) 적용 범위 밖 — 앞의 두 과제와 같은 결의 예외지만 **필요 근거는 서로 다르다**(장식적 표시 기능 vs
안내의 본질적 기능 vs 진행 상태의 시각적 서사).

- **사람 형태 아님·얼굴/표정 없음·평면 선화**(앞의 두 과제와 같은 원칙 유지). SVG 직접 렌더(외부 이미지 없음).
- **무채색 봉오리로 시작**: 뒤 꽃잎(`.lk-petal-back` ×3)·받침(`.lk-pad`)·중심(`.lk-core`)·앞 꽃잎 4장
  모두 회색으로 그려진 '닫힌 봉오리' 상태. 이 회색 무채색 요소는 자극이 아니라 봉오리 몸통(진행 서사의 배경).
- **단계마다 꽃잎 1장씩 개화**: 앞 꽃잎 4장(`FRONT_ROT` −50/−17/17/50°)을 자애 4단계에서 왼→오 순서로
  하나씩 accent(fill-opacity .5)로 0.8s 페이드 — 바디스캔 부위 페이드인과 동일한 방식(누적: 4단계에 다 핌).
- **마지막 감사 단계 = 중심부 발광**: 새 요소 없이 이미 다 핀 꽃의 중심(`.lk-core`)만 accent(.72)로
  더 밝게 전환. "감사는 자애의 연장선".
- viewBox 240×210, 좌표는 코드 상수(`CX/CY`, `PETAL/PETAL_BACK`, `FRONT_ROT/BACK_ROT`).

## 단계 구성 (자동 진행, 총 5단계)
자애 4단계(나 자신 → 가까운 사람 → 잘 모르는 사람 → 어려운 사람) + 감사 전환 1단계. `STAGES` 배열.
세션 길이 선택(5/10/15분) → **인트로(안착)·아웃트로(마무리) 예약 시간을 뺀 나머지를 5단계에 균등 분배**
(`buildSegments`: perStage = (sessionMs − INTRO_MS − OUTRO_MS) / 5). 세그먼트 = [인트로, 5단계, 아웃트로].

### ★ 4단계(어려운 사람) = 건너뛰기 허용
전통적 자애명상에서도 난이도가 높은 단계라 **강제하지 않는다**. 안내 문구 자체에 "어렵다면 지금은
건너뛰어도 괜찮습니다"를 넣고, 4단계 세그먼트에서만 **건너뛰기 버튼**(`.lk-skip`)을 노출한다(다른
단계에선 `hidden`). 누르면 진행 중 TTS 를 `cancel` 하고 즉시 다음(감사) 단계로(`skipSegment`).
이 프로젝트의 "부담 주지 않기" 원칙과 일치. (부주의맹의 "리셋 없음=정직함"과는 다른 종류의 배려지만
같은 결의 원칙.)

## ★ 안내 전달 — TTS(목소리) + 텍스트 동시 (바디스캔 엔진 그대로 재사용)
- `speechSynthesis`(Web Speech API)로 각 단계 문구를 읽어주고, **동시에 화면에도 같은 문구를 항상 텍스트로
  표시**(`.lk-guide`, aria-live polite).
- **다음 단계 전환 = TTS `onend` 와 단계별 최소 시간 중 '더 늦은 쪽'.** `cur.minDone`·`cur.ttsDone` 두
  플래그, `maybeAdvance()` 는 둘 다 true 일 때만 다음으로(최소 시간 backstop, TTS 가 더 길면 TTS 종료까지 대기).
- **폴백(에러 아님)**: TTS 미지원·해당 언어 음성 없음·음소거면 `speak()` 가 false → 즉시 `cur.ttsDone=true`
  로 두어 최소 시간만으로 자동 진행. try/catch 로 조용히 폴백(헤드리스=음성 없음이 이 경로를 검증).
- **★ TTS_CAP_MS 안전 상한을 처음부터 포함**(바디스캔에서 도입한 원칙): 말하기 시작했는데 `onend` 가
  유실되는 브라우저 특성 대비, 상한 안에 안 오면 강제로 '끝난 것'으로 처리해 세션이 멈추지 않게 한다.
  정상 브라우저에선 onend 가 최소시간보다 먼저 와 이 상한은 닿지 않는다('더 늦은 쪽' 규칙 유지).
- **첫 발화는 길이선택 버튼 클릭 핸들러 안에서 트리거**(render 의 session 분기 `playSegment(0)`,
  이 호출 스택이 곧 클릭 제스처 — 자동재생류 정책 회피). rate 0.88(명상 톤), pitch 기본.
- **음소거 토글**(🔔/🔇): 기본 켜짐, 세션마다 초기화(저장 안 함). 음소거 시 즉시 `cancel`+`ttsDone`으로 진행.
- **스테일 콜백 방지**: `segToken` 세그먼트마다 증가+모든 콜백 토큰 검사. 건너뛰기도 `playSegment` 가
  토큰을 올려 스킵된 세그먼트의 대기 콜백을 무효화한다.
- **★ 음질은 실제 브라우저에서 사용자가 직접 듣고 최종 판단**(바디스캔과 동일 — 소리 자체는 헤드리스로 검증 불가).

## 화면 (4언어)
1) 안내(자애·감사 소개·앉거나 누워도 됨·마음속으로만·판정 없음) → 2) 세션 길이 선택(5/10/15분, 이 클릭이
첫 발화 트리거) → 3) 세션(연꽃[단계별 개화] + 안내 텍스트 + 4단계 건너뛰기 버튼 + 얇은 진행바 + 음소거
토글, **언어바는 세션 중 숨김** — 앞 두 과제와 동일 근거) → 4) **완료 확인**(총 시간·거쳐온 단계 5).

## 이번엔 채택하지 않은 것 (기록)
- **일주일치 기록 보기**: 게임화 금지(스트릭/기록 금지)와 충돌 → 채택 안 함. 실천 4개 완료 후 프로젝트
  전체 차원에서 재검토(단발성 예외로 조용히 넣지 않음).
- **음성 입력(STT)·텍스트 입력**: 기술 리스크 대비 실익 낮고 입력 부담 제거 취지 → 전부 "마음속으로만"으로 결정.

변화)/**4단계 건너뛰기 버튼 노출·클릭 동작**/음소거 토글/**TTS 미지원·voice 없음 폴백(헤드리스가 이 경로를
검증)**/자동 종료→완료 화면(단계 5)/입력창 등 자기보고 UI 부재/JS 에러 0. `?qa=1` 이면 각 단계 최소 시간

결정론적으로 테스트.
